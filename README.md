# WatchTower Tester

A browser-based end-to-end test harness for the WatchTower monitoring app. It lets you fire real error and performance events from a live browser page into the WatchTower backend, so you can verify the full pipeline without needing a separate monitored application.

The hosted tester is available at: [https://cse110-sp26-group08.github.io/WatchTower-Tester/](https://cse110-sp26-group08.github.io/WatchTower-Tester/)

You can also clone this repo and open `index.html` directly in a browser — no build step or server required.

## Prerequisites

- You have an API key generated by WatchTower. If you do not have one go to [WatchTower](https://watchtower.team7ate92.workers.dev), login and create an app. You can find the API key in the app settings.

## Quickstart

1. Open the [hosted tester](https://cse110-sp26-group08.github.io/WatchTower-Tester/) (or `index.html` locally).
2. Paste your API key into the **API key** field.
3. Click **Load script tag** — the status badge in the top-right turns green when ready.
4. Click any test button.
5. Watch the **Network Log** at the bottom of the page.

A successful event shows a `201 Created` response in the log. The API key is saved in localStorage so you do not need to re-enter it after a reload.

## Test Buttons

### Errors

These buttons throw JavaScript errors that are captured and forwarded to `/api/events/error`.

| Button | What it throws |
|---|---|
| Manual error | `Error` |
| Runtime error | `TypeError` |
| Promise rejection | `ReferenceError` (unhandled) |

**More error types** (click to expand):

| Button | What it throws |
|---|---|
| SyntaxError | `SyntaxError` |
| RangeError | `RangeError` |
| Network error | `Error("network request failed…")` |
| Timeout error | `Error("operation timeout…")` |
| Throw custom | `Error` with your own message |

> The red errors in Chrome DevTools are expected — errors are intentionally thrown so WatchTower can capture them.

### Performance

| Button | What it does |
|---|---|
| Page performance | Page-load metrics are sent automatically on script load. Reload and re-load the script tag to retest. |
| Fetch slow URL | Fetches a 12 MB public image to generate a fetch-latency event at `/api/events/performance`. |

**More fetch targets** (click to expand):

| Button | Target | Purpose |
|---|---|---|
| Fetch fast URL | jsonplaceholder `/posts/1` (~1 KB) | Tests the low-latency happy path |
| Fetch 21 MB image | Large Wikimedia image | Tests a heavier slow-network scenario |
| Fetch failing URL | `httpstat.us/500` | Tests error handling on a failed fetch |
| Fetch custom | URL you type in | Test your own endpoints or APIs |

### Direct Smoke Tests

These two buttons POST a minimal payload straight to the WatchTower API, bypassing the script tag entirely. Use them to confirm the backend is reachable and accepting requests on its own.

| Button | Endpoint hit |
|---|---|
| Direct error POST | `POST /api/events/error` |
| Direct performance POST | `POST /api/events/performance` |

## Network Log

Every outbound request is printed to the dark log panel, newest entry first. Each line shows:

```
[HH:MM:SS AM/PM] POST https://watchtower.team7ate92.workers.dev/api/events/error → 201 Created
{"message":"ok"}
```

Click **Copy log** to copy the full log to your clipboard.

Color coding:

- Green — successful response (`2xx`)
- Yellow — non-fatal warning or non-`2xx` HTTP response
- Red — network failure or backend error
- Blue — outgoing request or informational message

## Notes

- The **Reload page** button clears the script tag and resets the page. Use it when you want to retest page-load performance from scratch.
